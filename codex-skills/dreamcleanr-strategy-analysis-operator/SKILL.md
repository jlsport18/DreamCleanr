# DreamCleanr Strategy Analysis Operator

## Overview

Package DreamCleanr's master strategy prompt into a reusable operating skill so product, moat, monetization, GTM, and architecture analysis can be refreshed without drifting from repo truth.

## Use This Skill

- Read [../../DREAMCLEANR_MASTER_STRATEGY.md](../../DREAMCLEANR_MASTER_STRATEGY.md), [../../MARKET_STRATEGY.md](../../MARKET_STRATEGY.md), [../../MONETIZATION_PLAN.md](../../MONETIZATION_PLAN.md), and [../../ROADMAP.md](../../ROADMAP.md) first.
- Read [../../COMMERCIAL_BACKEND_ARCHITECTURE.md](../../COMMERCIAL_BACKEND_ARCHITECTURE.md), [../../FIGMA_READY_UI_BRIEF.md](../../FIGMA_READY_UI_BRIEF.md), [../../GROWTH_LAUNCH_PLAYBOOK.md](../../GROWTH_LAUNCH_PLAYBOOK.md), and [../../ONBOARDING_CONVERSION_SPEC.md](../../ONBOARDING_CONVERSION_SPEC.md) when the strategy refresh touches future-commercial incubation, native-shell design, or launch systems.
- Read [references/codex-master-prompt.md](references/codex-master-prompt.md) before drafting or refreshing the canonical analysis.
- Use [references/analysis-checklist.md](references/analysis-checklist.md) as the final pass before publishing strategy changes.
- Preserve current DreamCleanr truth:
  - Community free
  - Pro later as a one-time premium macOS shell
  - Team later
  - local-first and GitHub-first remain canonical

## Core Workflow

1. Ground the analysis in the current codebase and shipped behavior.
2. Run the 12-section master-prompt structure against current DreamCleanr truth.
3. Distinguish shipped strengths, planned detectors, future native-shell work, and future-commercial incubation clearly.
4. Route design depth, launch systems, and MRR planning into their modular docs instead of collapsing them into a single founder memo.
5. Keep [../../DREAMCLEANR_MASTER_STRATEGY.md](../../DREAMCLEANR_MASTER_STRATEGY.md) canonical and shorten rollups instead of letting them drift into competing strategies.

## Analysis Rules

- Lead with current strengths: Docker, Claude, Codex, safe cache cleanup, receipts, MCP, scheduling.
- Keep current gaps explicit: Python, Node, Ollama, LM Studio, Hugging Face, Git/LFS, IDE intelligence, project-aware delete logic.
- Do not quietly replace the current business model with SaaS-first subscriptions.
- Treat backend, Stripe, auth, analytics, and MRR systems as future-only unless the canonical strategy is intentionally changed.
- Keep SwiftUI shell direction canonical even when future backend or web surfaces are being planned.

## Good Deliverables

- master strategy refreshes
- moat and opportunity maps
- concise rollups for market, roadmap, and monetization
- alignment passes across modular docs for commercial incubation, native-shell planning, and grounded growth systems
- decision-complete top actions and growth moves

## Avoid

- investor-style hype disconnected from the repo
- generic cleaner positioning
- claims that planned detector support or live subscriptions already exist
